# דו"ח שינויי קוד יומי - 21/07/2026

דו"ח זה הופק אוטומטית על ידי מערכת ה-AI לסקירת קוד לילית. הדו"ח סוקר 2 קומיטים (Commits) מרכזיים שבוצעו במהלך היממה האחרונה.&rlm;

## 1. סקירת שינויים עיקריים

ביממה האחרונה התבצעה התקדמות משמעותית בפיתוח ממשק המשתמש של לוח הבקרה, תהליך הקליטה (Onboarding) ומודול ניהול ההכנסות וההוצאות באפליקציה.&rlm;

### ממשק משתמש וצד לקוח (Front-end)&rlm;
- **תהליך הרשמה וקליטה (Onboarding)**: הוקם תהליך Onboarding חדש הכולל רכיבים ייעודיים (`OnboardingClient.tsx`, `OnboardingShell.tsx`) כדי להוביל את המשתמש במסלול הגדרת החשבון והשירותים.&rlm;
- **לוח בקרה וניווט (Dashboard Shell & Navigation)**: שופר ממשק משתמש מרכזי עם עדכונים ב-`DashboardShell.tsx` ו-`DashboardClient.tsx`. נוספו רכיבי ניווט מתקדמים כמו `MosaicMenuModal.tsx`, `RhombusMenu.tsx` ותפריט הגדרות `CenterSettingsModal.tsx`.&rlm;
- **ניהול הכנסות והוצאות**: נוספו מודולים מלאים לניהול הכנסות והוצאות (`IncomesDashboardClient.tsx`, `ExpensesDashboardClient.tsx`). נוספו מודלים להוספת הכנסה (`AddIncomeModal.tsx`), להוספת הוצאה (`AddExpenseModal.tsx`) ולהיסטוריית הכנסות/הוצאות.&rlm;
- **שירותי חברה**: שולבו כרטיסיות מיתוג ושירותי חברה (`BrandingTab.tsx`, `CompanyServicesSection.tsx`, `ServicesDashboardClient.tsx`).&rlm;

### צד שרת ונתונים (Back-end, Database & Storage)&rlm;
- **פעולות שרת (Server Actions)**: נוספו שירותי API לניהול נתוני ההכנסות (`src/features/incomes/actions.ts`) כולל הגדרות Typescript מסודרות (`types.ts`). כמו כן, בוצעו עדכונים ב-`actions.ts` של מודול ה-AI.&rlm;
- **לוגים ותיעוד**: מערכת הלוגים `debug-crm.log` המשיכה לתעד אירועים מהמערכת.&rlm;

---

## 2. חלוקה ארכיטקטונית

* **צד לקוח (Front-end)**: זהו מוקד השינויים העיקרי היום. נעשה שימוש נרחב ברכיבי React ליצירת תהליכים מורכבים כמו Onboarding וניהול פיננסי. שילוב המודלים הקופצים (Modals) והניווט המרכזי מבוסס על TailwindCSS וגישת ה-App Router של Next.js.&rlm;
* **צד שרת (Back-end)**: הרחבת יכולות ה-Server Actions מאפשרת לממשקי התצוגה לשמור נתונים ישירות בצד שרת בצורה מאובטחת בלי לחשוף API ציבורי לניהול הכנסות והגדרות AI.&rlm;
* **מסד נתונים (Database)**: פעולות ההכנסות רומזות על התממשקות למסד הנתונים ליצירת רשומות, עדכונן ושליפת ההיסטוריה הפיננסית של המשתמש.&rlm;
* **אחסון (Storage)**: קבצי הלוגים ו-`daily_changes.json` ממשיכים לפעול כרשומות זמניות ובקרה של האפליקציה.&rlm;

---

## 3. הערכת יציבות ארכיטקטונית

1. **תאימות ממשקים**: הקוד שנוסף (למשל מודולי ההכנסות וההוצאות) מפריד בצורה תקינה בין רכיבי תצוגה עיוורים (Dumb Components כמו הטפסים והרשימות) לרכיבי שליטה שמנהלים את הסטייט (Client Dashboards).&rlm;
2. **אבטחה וניתוב**: השילוב של `auth-redirect/page.tsx` ותהליך ה-Onboarding מעיד על שיפור אבטחת הגישה וחיוב משתמשים להשלים פרופיל לפני כניסה ללוח הבקרה הראשי. חשוב לוודא שה-Middleware מכסה את כל נתיבי ה-Dashboard כדי למנוע מעקפים.&rlm;
3. **יציבות**: הפרויקט שומר על ארכיטקטורה יציבה. הוספת תפריטי ניווט מיוחדים (Mosaic, Rhombus) עשויה להשפיע על חווית המשתמש במובייל, ויש לבדוק את הרספונסיביות שלהם.&rlm;

---

## 4. המלצות וצעדי פיתוח למחר

1. **בדיקות רספונסיביות (UI)**: הממשקים החדשים (תפריטי צורות ומודלים של הכנסות/הוצאות) דורשים בדיקה מעמיקה במכשירים ניידים כדי להבטיח שלא נוצרו גלילות מיותרות או שבירות מסך.&rlm;
2. **אינטגרציה של קבלות ידניות**: הרכיב `KesherManualReceiptsForm.tsx` נוסף אך חשוב לוודא חיבור תקין למערכת Kesher API ושמירה עקבית במסד הנתונים מול עסקאות שלא תועדו אוטומטית.&rlm;
3. **אכיפת State ב-Onboarding**: לוודא שיש חסימה שרתית לפעולות באפליקציה למשתמשים שלא סיימו את שלבי ה-Onboarding בהצלחה.&rlm;
4. **עמידה ב-RTL ונגישות**: כמו תמיד, להקפיד על הוספת `&rlm;` ותמיכת `dir="rtl"` בכל המודלים והתפריטים שנוספו עכשיו כדי לא לשבור את כיוון סימני הפיסוק בטקסטים המשלבים אנגלית ועברית.&rlm;
